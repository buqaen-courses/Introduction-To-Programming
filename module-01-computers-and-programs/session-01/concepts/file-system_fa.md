# سیستم فایل: سازماندهی داده کامپیوتر

## به زبان ساده

**چه چیزی یاد می‌گیرید:** چگونه کامپیوترها فایل‌ها را ذخیره، سازماندهی و پیدا می‌کنند. سیستم فایل را به عنوان یک کتابخانه دیجیتال در نظر بگیرید—چیزی که همه چیز مرتب نگه می‌دارد، موقعیت‌ها را پیگیری می‌کند، و به شما اجازه می‌دهد چیزی را که نیاز دارید پیدا کنید. شما درباره پوشه‌ها (دایرکتوری‌ها)، مسیرهای فایل، انواع فایل، و چگونگی نگهداری اطلاعات توسط سیستم فایل خواهید آموخت.

**نکته برای تازه‌کاران:** هنگام برنامه‌نویسی، با فایل‌ها و پوشه‌ها زیاد سروکار خواهید داشت. درک مسیرها و ساختار سیستم فایل برای خواندن/نوشتن فایل‌ها، سازماندهی پروژه‌ها و اجتناب از «فایل یافت نشد» ضروری است.

## سیستم‌های فایل مختلف

```
┌─────────────────────────────────────────────────────────────────────┐
│                    انواع سیستم فایل                                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  ویندوز:                                                             │
│  ═════════════════════════════════════════════════════════==         │
│                                                                      │
│  NTFS (New Technology File System)                                 │
│  • فایل‌سیستم پیش‌فرض مدرن ویندوز                                   │
│  • اندازه فایل: تا ۱۶ exabytes (تئوری)                               │
│  • مجوزهای پیشرفته، رمزگذاری، فشرده‌سازی                            │
│  • journaling (بازیابی از خرابی)                                     │
│                                                                      │
│  FAT32 (File Allocation Table 32-bit)                              │
│  • قدیمی اما سازگار گسترده                                          │
│  • محدودیت: فایل حداکثر ۴GB                                         │
│  • هنوز روی درایوهای USB/SD استفاده می‌شود                          │
│                                                                      │
│  exFAT (Extended File Allocation Table)                            │
│  • جانشین مدرن FAT32                                                │
│  • فایل‌های بزرگ، سربار کم                                           │
│  • عالی برای درایوهای خارجی                                          │
│                                                                      │
│  ─────────────────────────────────────────────────────────────       │
│                                                                      │
│  macOS:                                                              │
│  ═════════════════════════════════════════════════════════==         │
│                                                                      │
│  APFS (Apple File System)                                            │
│  • فایل‌سیستم پیش‌فرض مدرن macOS/iOS                                 │
│  • snapshot‌ها، فضای اشتراکی، رمزگذاری                               │
│  • بهینه‌سازی شده برای SSD‌ها                                          │
│  • کپی‌نوشت (کپی کارآمد فایل‌ها)                                     │
│                                                                      │
│  HFS+ (Hierarchical File System Plus)                              │
│  • قدیمی، در سیستم‌های قدیمی‌تر macOS                                  │
│                                                                      │
│  ─────────────────────────────────────────────────────────────       │
│                                                                      │
│  لینوکس:                                                             │
│  ═════════════════════════════════════════════════════════==         │
│                                                                      │
│  ext4 (Fourth Extended Filesystem)                                   │
│  • فایل‌سیستم پیش‌فرض رایج‌ترین توزیع‌های لینوکس                     │
│  • journaling، فایل‌های بزرگ، دسترسی سریع                            │
│  • عالی برای سرورها و ایستگاه‌های کاری                               │
│                                                                      │
│  XFS                                                                 │
│  • عالی برای فایل‌های بسیار بزرگ                                      │
│  • رایج در محیط‌های سازمانی                                          │
│                                                                      │
│  Btrfs (B-tree filesystem)                                           │
│  • ویژگی‌های پیشرفته (snapshot، RAID، فشرده‌سازی)                    │
│  • در حال توسعه فعال، پایدارتر می‌شود                                │
│                                                                      │
│  ZFS (Zettabyte File System)                                         │
│  • نجات‌دهنده داده (کنترل یکپارچگی)                                  │
│  • snapshot، رمزگذاری، RAID                                           │
│  • سربار بالا اما قدرتمند                                            │
│                                                                      │
│  ─────────────────────────────────────────────────────────────       │
│                                                                      │
│  چرا انواع مختلف؟                                                   │
│  ═════════════════════════════════════════════════════════==         │
│                                                                      │
│  هر سیستم فایل ویژگی‌های مختلفی را اولویت‌بندی می‌کند:               │
│                                                                      │
│  سازگاری:  FAT32، exFAT، NTFS                                        │
│  عملکرد:   APFS، ext4، XFS                                           │
│  ویژگی‌ها: ZFS، Btrfs، APFS                                          │
│  قابلیت اعتماد: ZFS، ext4 (با journaling)                           │
│  سربار کم: FAT32، exFAT                                              │
│                                                                      │
│  ⚠️ درایوهای USB بین macOS، ویندوز، لینوکس اشتراک می‌کنند؟         │
│  → از exFAT برای سازگاری استفاده کنید                               │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

## پاسخ به سوالات رایج مبتدی‌ها

| سوال | پاسخ ساده |
|------|-----------|
| **"تفاوت بین مسیر مطلق و نسبی چیست؟"** | مطلق از ریشه شروع می‌شود (همیشه کار می‌کند)، نسبی از موقعیت فعلی شما (کوتاه‌تر، اما به موقعیت بستگی دارد). |
| **"چرا باید از پوشه‌ها استفاده کنم به جای اینکه همه چیز را در یکجا رها کنم؟"** | پوشه‌ها سازماندهی منطقی می‌کنند، یافتن فایل‌ها را آسان‌تر می‌کنند، به گروه‌بندی فایل‌های مرتبط کمک می‌کنند و جدا کردن نگرانی‌ها را ممکن می‌سازند. |
| **"افزونه فایل چقدر مهم است؟"** | افزونه یک راهنماست—معمولاً نشان می‌دهد چه نوع فایلی است—اما محتوای واقعی فایل را تغییر نمی‌دهد. سیستم برای شناسایی نوع محتوای واقعی را بررسی می‌کند. |
| **"چرا نمی‌توانم کاراکترهای خاصی در نام فایل داشته باشم؟"** | کاراکترهایی مانند `/ \ : * ? " < >` |` برای سیستم عامل معنای خاص دارند. جلوگیری از آنها از تداخل با دستورات/عملیات جلوگیری می‌کند. |
| **"چه اتفاقی برای فایل‌ها می‌افتد وقتی حذف می‌شوند؟"** | فایل حذف نمی‌شود—تنها ارجاع در سیستم فایل حذف می‌شود. بلوک‌های داده در دیسک باقی می‌مانند تا زمانی که رونویسی شوند (این امکان بازیابی داده را فراهم می‌کند). |
| **"چرا یک پوشه می‌تواند هم پوشه و هم فایل داشته باشد؟"** | پوشه‌ها فقط میانبرهای منطقی هستند—نه ظروف فیزیکی. فایل‌ها در بلوک‌های دیسک پراکنده شده‌اند و سیستم فاید پیگیری می‌کند کدام پوشه به کدام بلوک ارجاع می‌دهد. |

## بررسی سریع (درک خود را بسنجید)

سعی کنید قبل از ادامه، این‌ها را با کلمات خودتان پاسخ دهید:

۱. **تفاوت بین مسیر مطلق و نسبی چیست و چه زمانی باید از هر کدام استفاده کنید؟**
   <details>
   <summary>برای پاسخ کلیک کنید</summary>
   مسیر مطلق از ریشه سیستم فایل شروع می‌شود (مثل `/home/user/file.txt` یا `C:\Users\User\file.txt`) و همیشه به همان موقعیت اشاره می‌کند صرف‌نظر از اینکه کجا هستید. مسیر نسبی از موقعیت فعلی شما شروع می‌شود (مثل `../file.txt`) و به موقعیت بستگی دارد. از مطلق برای پیکربندی، مسیرهای وابستگی، یا موقعیت‌هایی که باید در هر جا کار کنند استفاده کنید. از نسبی در پروژه‌ها، برای واردات، یا زمانی که می‌خواهید کد قابل حمل بین محیط‌ها باشد استفاده کنید.
   </details>

۲. **سیستم فایل چگونه می‌داند یک فایل کجاست وقتی پوشه‌ها «ظروف فیزیکی» نیستند؟**
   <details>
   <summary>برای پاسخ کلیک کنید</summary>
   سیستم فایل از یک «جدول تخصیص فایل» یا ساختار مشابه استفاده می‌کند که بلوک‌های دیسک فیزیکی را به فایل‌های منطقی نگاشت می‌کند. وقتی یک فایل را باز می‌کنید، سیستم فایل موقعیت بلوک‌های دیسک که حاوی داده هستند را جستجو می‌کند و آن بلوک‌ها را می‌خواند. پوشه‌ها فقط لیست‌هایی از نام فایل‌ها به شماره بلوک هستند.
   </details>

۳. **چرا افزونه فایل (.txt، .py، .jpg) نمی‌تواند به طور مطمئن نوع فایل را مشخص کند؟**
   <details>
   <summary>برای پاسخ کلیک کنید</summary>
   افزونه فایل فقط بخشی از نام است—می‌توانید هر فایل را به هر چیزی تغییر نام دهید. محتوای واقعی فایل تعیین می‌کند چیست. سیستم عامل‌ها با خواندن بایت‌های اولیه (محدوده جادویی) تعیین می‌کنند. به عنوان مثال، فایل‌های PNG با `‰PNG` شروع می‌شوند، صرف‌نظر از افزونه. یک فایل می‌تواند `.txt` باشد اما در واقع یک تصویر باشد.
   </details>

۴. **اگر فایلی را حذف کنید چه اتفاقی می‌افتد و چرا گاهی می‌توان فایل‌های حذف شده را بازیابی کرد؟**
   <details>
   <summary>برای پاسخ کلیک کنید</summary>
   حذف کردن فایل داده را حذف نمی‌کند—فقط ورودی سیستم فایل را حذف می‌کند که بلوک‌های دیسک را به عنوان «در حال استفاده» علامت‌گذاری می‌کند. بلوک‌های داده باقی می‌مانند تا زمانی که سیستم فایل آنها را برای یک فایل جدید رزرو کند و رونویسی شوند. نرم‌افزار بازیابی می‌تواند بلوک‌های رها شده را اسکن کند و سعی کند فایل را بازسازی کند. به همین دلیل است که حذف ایمن (رونویسی) برای داده‌های حساس وجود دارد.
   </details>

۵. **NTFS، APFS، و ext4 چه تفاوت‌هایی دارند و چرا سیستم عامل‌های مختلف از سیستم‌های فایل مختلف استفاده می‌کنند؟**
   <details>
   <summary>برای پاسخ کلیک کنید</summary>
   اینها پیاده‌سازی‌های مختلف سیستم فایل با ویژگی‌های مختلف هستند. NTFS (ویندوز) روی journaling، رمزگذاری، و ویژگی‌های سازمانی متمرکز است. APFS (macOS) برای SSDها بهینه شده، کپی‌نوشت (snapshotها) و رمزگذاری دارد. ext4 (لینوکس) عملکرد خوب، قابلیت اعتماد، و سازگاری گسترده را متعادل می‌کند. هر سیستم عامل سیستم فایل را بهینه کرده است تا با معماری و موارد استفاده هدف خود مطابقت داشته باشد.
   </details>

## تمرین‌های عملی

### تمرین ۱: مسیرها را ترسیم کنید

با توجه به این ساختار:
```
/home/ali/projects/webapp/
├── src/
│   └── main.py
├── tests/
│   └── test_main.py
└── data/
    └── users.json
```

چه مسیرهایی را برای دسترسی به این فایل‌ها از موقعیت `/home/ali` استفاده می‌کنید؟
- main.py (نسبی): ____________
- test_main.py (مطلق): ____________
- users.json (نسبی با `..`): ____________

### تمرین ۲: ابرداده را بخوانید

یک فایل متنی ایجاد کنید، چند بار آن را ویرایش کنید، سپس با استفاده از این دستورات اطلاعات آن را بررسی کنید:

```bash
# این را در ترمینال خود امتحان کنید
ls -l your_file.txt
stat your_file.txt
```

چه اطلاعاتی می‌بینید؟ آیا تاریخ اصلاح پس از ویرایش تغییر کرده است؟

### تمرین ۳: جستجو در فایل‌ها

چگونه می‌توانید:
۱. همه فایل‌های پایتون (`.py`) را در یک پروژه پیدا کنید؟
۲. همه فایل‌ها را که بزرگ‌تر از ۱۰ مگابایت هستند پیدا کنید؟
۳. فایل‌هایی را که در ۷ روز گذشته تغییر کرده‌اند پیدا کنید؟

<details>
<summary>راهنمایی‌ها</summary>

۱. `find . -name "*.py"`
۲. `find . -size +10M`
۳. `find . -mtime -7`

</details>

### تمرین ۴: طراحی پروژه

یک پروژه فرضی را برای یک برنامه فهرست کار (todo app) طراحی کنید. چه پوشه‌ها و فایل‌هایی ایجاد می‌کنید؟ چه فایل‌ها در هر پوشه قرار می‌گیرند؟

## مطالعه و کاوش بیشتر

- **بیشتر بیاموزید:** راجع به فایل‌سیستم‌های توزیع‌شده (NFS، SMB) برای اشتراک‌گذاری شبکه
- **کاوش کنید:** لینوکس (inodes، پیوندهای سخت/نمادین) برای درک سیستم فایل عمیق‌تر
- **مطالعه کنید:** پیکربندی و مدیریت مجوزهای فایل (chmod، chown، ACLها)
- **امتحان کنید:** یک پروژه با ساختار پوشه مناسب راه‌اندازی کنید
- **بازی کنید:** تمرین دستورات ترمینال برای عملیات فایل
- **گام بعدی:** یادگیری خواندن/نوشتن فایل‌ها از برنامه خود
